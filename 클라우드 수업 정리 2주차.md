## 1. AWS 핵심 서비스 개요

### 컴퓨팅 서비스

- **EC2 (Elastic Compute Cloud)**: 가상 서버, 온디맨드/스팟 인스턴스
- **Lambda**: 서버리스 컴퓨팅, 이벤트 기반 실행

### 스토리지 서비스

- **S3 (Simple Storage Service)**: 객체 스토리지, 정적 웹 호스팅
- **EBS (Elastic Block Store)**: EC2 인스턴스를 위한 블록 스토리지

### 네트워크 및 콘텐츠 전송

- **VPC (Virtual Private Cloud)**: AWS 내 네트워크 구성
- **Route 53**: 도메인 네임 서비스 (DNS)
- **CloudFront**: CDN (콘텐츠 전송 네트워크)

### 데이터베이스 서비스

- **RDS (Relational Database Service)**: 관리형 데이터베이스 (MySQL, PostgreSQL, MariaDB 등)
- **DynamoDB**: NoSQL, 키-값 저장소

### 보안 및 접근 관리

- **IAM (Identity and Access Management)**: 사용자 및 권한 관리
- **Security Group (보안 그룹)**: 인바운드 및 아웃바운드 트래픽 제어

## 2. AWS 구조

### 리전 (Region)

- AWS 서비스가 제공되는 **서버의 물리적 위치**
- 전 세계에 분산되어 있으며 동남아, 유럽, 북아메리카 등으로 구분
- 각 리전에는 고유한 코드가 부여됨
    - 예) 서울: `ap-northeast-2`
    - 예) 미국 동부(버지니아 북부): `us-east-1` → **모든 서비스가 최초로 출시되는 리전**
- 리전별로 사용 가능한 서비스가 다를 수 있음

### 리전 선택 시 고려 사항

| 항목 | 설명 |
| --- | --- |
| 지연 속도 | 사용자와 가까운 리전 선택 (예: 북미 서비스 → 북미 리전) |
| 법률/규정 | 개인정보는 해당 국가 서버에 저장해야 하는 경우 있음 |
| 사용 가능 서비스 | 리전별 서비스 출시 시기가 다름 |

> **us-east-1**: 모든 AWS 서비스가 최초로 서비스되는 리전. 빌링, CloudFront 등 글로벌 서비스의 기준 리전.
> 

---

### 가용 영역 (Availability Zone, AZ)

- 리전의 **하부 단위**
- 하나의 리전은 **2개 이상의 AZ**로 구성
- 1개 이상의 물리 데이터 센터를 묶은 **논리적 데이터 센터**

### AZ의 특징

- AZ 간 연결은 매우 빠른 **전용 네트워크**로 구성
- 물리적으로 일정 거리(수 km 이상) 떨어져 있음
- 모든 AZ는 서로 **100km 이내**에 위치 → 재해 대비 및 보안 목적

### AZ 위치의 랜덤성

- 계정마다 AZ 코드와 실제 물리 위치가 다르게 매핑됨
- 예) 계정 A의 `us-east-1a` ≠ 계정 B의 `us-east-1a`
- 이유: 보안 이슈 방지 + 특정 AZ로의 트래픽 쏠림 현상 방지

---

### 엣지 로케이션 (Edge Location)

- CloudFront(CDN), Global Accelerator 등의 서비스를 **가장 빠르게 제공**하기 위한 캐싱 거점
- 전 세계 수백 곳에 분산 배치 (리전보다 훨씬 많음)
- **출장소** 개념으로 이해하면 쉬움

### CDN (Contents Delivery Network)

- 콘텐츠를 사용자와 **가까운 위치에서 빠르게 전송**하는 기술
- 엣지 로케이션이 없다면 → 느린 응답 속도, 스트리밍 불가
- 엣지 로케이션을 통해 → **빠른 응답 + 부하 분산** 가능

---

### 글로벌 서비스 vs 리전 서비스

| 구분 | 설명 | 예시 |
| --- | --- | --- |
| 글로벌 서비스 | 전 세계 모든 인프라가 공유 | IAM, Route 53, WAF, Billing |
| 리전 서비스 | 특정 리전 기반으로 데이터 및 서비스 제공 | EC2, RDS, S3 등 대부분의 서비스 |

---

### ARN (Amazon Resource Names)

- AWS 리소스에 부여되는 **고유 식별자**
- 형식: `arn:aws:{서비스}:{리전}:{계정ID}:{리소스}`
- 예) `arn:aws:ec2:us-east-1:123456789012:vpc/vpc-0e9801d129EXAMPLE`

## 3. AWS 서비스 구조 (네트워크 관점)

- **VPC**는 원칙적으로 퍼블릭 인터넷에서 직접 접근 불가
- AWS 내부 서비스라도 원칙적으로 퍼블릭 인터넷을 통해 접근
- **인터넷 게이트웨이(Internet Gateway)**를 통해 외부와 통신

---

## 4. VPC (Virtual Private Cloud)

### 개요

- AWS 계정 전용 **가상 네트워크** (가상 데이터 센터)
- 외부와 **격리된 사설망** 구축이 목적
- **리전 단위**로 생성

### VPC 구성 요소

| 구성 요소 | 설명 |
| --- | --- |
| **Subnet** | VPC를 특정 IP 범위로 나눈 논리적 단위 |
| **Route Table** | 네트워크 트래픽을 어디로 보낼지 명시한 규칙 테이블 |
| **Internet Gateway (IGW)** | VPC 리소스의 인터넷 통신을 활성화하는 게이트웨이 |
| **NAT Gateway** | Private Subnet에서 인터넷 통신을 가능하게 하는 주소 변환 게이트웨이 |
| **VPC Endpoint** | NAT/IGW 없이 AWS 서비스에 비공개로 연결하는 서비스 |

### VPC 아키텍처 구성 예시

- Public Subnet: 외부 접근 가능 (EC2 + EBS + AMI + Security Group)
- Private Subnet: 외부 접근 불가 (EC2 + Auto Scaling Group + SG)
- 각 AZ에 Public/Private Subnet을 나누어 **고가용성** 구성
- ELB(Elastic Load Balancer)로 트래픽 분산

## 5. IAM (Identity and Access Management)

### 개요

- AWS 리소스에 대한 **접근을 안전하게 제어**하는 서비스
- **누가, 어떤 서비스에, 어떤 권한으로** 접근할 수 있는지 관리

### IAM 구성 요소

| 요소 | 설명 |
| --- | --- |
| **User (사용자)** | AWS 리소스에 접근하는 개별 계정 |
| **Group (그룹)** | 여러 사용자에게 동일한 권한을 부여하는 그룹 |
| **Role (역할)** | 특정 권한을 임시로 부여하는 역할 (서비스 간 권한 위임 가능) |
| **Policy (정책)** | JSON 형식으로 작성된 접근 권한 규칙 |
| **MFA** | 다중 인증(OTP 등)으로 보안 강화 |

### Policy 예시 (특정 S3 버킷 읽기 권한)

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket-name/*"
    }
  ]
}
```

### IAM 주요 기능

- AWS 서비스별 세밀한 권한 제어 (EC2, S3, RDS 등)
- IAM Role을 통한 서비스 간 권한 위임 (예: EC2 → S3 읽기 허용)
- MFA 적용으로 로그인 보안 강화
- **루트 계정 보호**: 루트 계정은 모든 권한을 가지므로, 일반 사용에는 IAM 사용자 계정을 따로 생성해서 사용 권장

### ⚠️ IAM 사용 시 주의사항

- 루트 계정(root user)은 최소한으로만 사용
- **최소 권한 원칙(Least Privilege Principle)** 준수 → 꼭 필요한 권한만 부여
- IAM 액세스 키(Access Key)를 GitHub 등 공개 저장소에 절대 업로드 금지
- 잘못된 IAM 정책 설정은 심각한 보안 사고로 이어질 수 있음
