# AWS EC2 자동 모니터링 & CloudWatch 노트

## 1. 자동 모니터링 경보 생성

### 경보 생성 시작

- CloudWatch 콘솔 이동
- **경보 없음** 화면에서 **경보 생성** 클릭
- **지표 선택** 클릭

### 지표 설정 (단계 1: 지표 및 조건 지정)

- 검색창에 `CPUUtilization` 입력
- **인스턴스별 지표** 선택
- 모니터링할 내 인스턴스에 체크
- 옵션 탭에서 그래프 유형 → **누적 면적** 선택
- **지표 선택** 클릭

### 조건 설정

| 항목 | 값 |
| --- | --- |
| 기간 | 1분 |
| 임계값 유형 | 정적 |
| 조건 | 보다 큼 (>) |
| 임계값 | 30 |
- **추가 구성** 펼치기
- 누락된 데이터 처리 → **누락된 데이터를 양호(임계값 위반 안 함)으로 처리** 선택
- **다음** 클릭

---

## 2. 알림(SNS) 설정 (단계 2: 작업 구성)

### SNS 주제 생성

- **새 주제 생성** 선택
- 주제 이름: `High_CPU_Alarm`
- 알림 수신 이메일 엔드포인트 입력 (예: `본인이메일@도메인`)
- **주제 생성** 클릭

### EC2 작업 추가 (선택)

- **EC2 작업 추가** 클릭
- 경보 상태 트리거: **경보 상태** 선택
- 작업 선택: **이 인스턴스 재부팅** (또는 중지/종료 등)
- **다음** 클릭

### 이메일 구독 승인

> AWS에서 입력한 이메일로 구독 확인 메일이 발송됨
> 
> 
> 메일의 **Confirm subscription** 링크 클릭 → 승인 완료
>

## 3. 경보 이름 설정 (단계 3: 이름 및 설명 추가)

- 경보 이름: `High_CPU_Alarm`
- 설명은 선택 사항 (마크다운 형식 지원, 단 콘솔에서만 적용)
- **다음** → **경보 생성** 클릭

## 4. EC2에서 CPU 부하 발생시키기

EC2에 SSH(PuTTY)로 접속 후 아래 명령 실행:

```bash
# 1. Amazon Linux 추가 리포지토리 활성화
amazon-linux-extras install -y epel

# 2. stress-ng 설치 (CPU 부하 툴)
yum install -y stress-ng

# 3. CPU 부하 발생 (1개 CPU, 10분 동안)
stress-ng --cpu 1 --timeout 10m --metrics --times
```

→ CPU 사용률이 임계값(30%)을 초과하면 **이메일 알림** 발송

## 5. 실습) 네트워크 경고 만들기

### EC2에서 iperf3 설치 및 네트워크 부하 발생

```bash
# iperf3 설치
sudo yum install -y iperf3

# 서버 모드 활성화
iperf3 -s &

# CPU + 네트워크 동시 부하 발생 (10분)
iperf3 -s & sleep 2 && stress-ng --cpu 1 --timeout 10m --metrics --times & iperf3 -c 127.0.0.1 -t 600 -b 100M
```

### CloudWatch 경보 설정값

| 항목 | 값 |
| --- | --- |
| 지표 | `NetworkOut` |
| 임계값 | `NetworkOut >= 20000` |
| 경보 이름 | `High_Network_Alarm` |

### 실습 순서

1. EC2에 iperf3 설치
2. 서버 모드 실행 후 네트워크 부하 발생
3. CloudWatch에서 `NetworkOut` 지표로 경보 생성
4. EC2 재기동
5. 이메일로 알람 수신 확인

## 6. CloudWatch 핵심 정리

| 기능 | 설명 |
| --- | --- |
| **Metrics (지표)** | CPU, 네트워크 등 숫자로 보는 상태 데이터 |
| **Logs (로그)** | 텍스트 형태의 시스템 기록 |
| **Alarm (알람)** | 임계값 초과 시 SNS 이메일 알림 |
| **Dashboard (대시보드)** | 모든 지표를 한 화면에서 시각화 |

> **CloudWatch = 클라우드 운영의 눈(Eye)**
> 
> 
> AWS 리소스를 실시간으로 관찰하고, 기록하고, 이상 시 자동으로 대응하는 모니터링 서비스

# AWS S3 (Simple Storage Service) 노트

---

## 1. S3란?

- AWS에서 EC2와 함께 가장 오래되고 기본이 되는 **객체 스토리지 서비스**
- 확장성, 내구성, 보안성이 뛰어난 스토리지
- **99.999999999% (11 9's)** 의 내구성 제공 → 데이터 손실 최소화
- 데이터 저장 공간 **거의 무제한**
- 전 세계 수백만 기업이 사용하는 핵심 AWS 서비스

---

## 2. S3 특징

| 특징 | 설명 |
| --- | --- |
| 객체 스토리지 | 파일 저장 전용 (블록 스토리지 EBS/EFS와 다름) → **애플리케이션 설치 불가** |
| 글로벌 서비스 | 서비스 자체는 글로벌, 데이터는 특정 리전에 저장 |
| 무제한 용량 | 단일 객체 0byte ~ **5TB** 저장 가능 |
| 고가용성 | 최소 **3개 이상**의 물리적으로 분리된 가용 영역(AZ)에 데이터 복제 |
| 접근 편의성 | OS 도움 없이 객체별 직접 접근 가능 |

---

## 3. 버킷(Bucket)과 객체(Object)

### 버킷

- S3에서 저장 공간을 구분하는 단위 (디렉터리 개념)
- **전 세계 고유 이름** 필요 → 리전과 무관하게 중복 불가
- 한번 생성 후 **이름 변경 불가**

### 객체

S3에 저장되는 기본 데이터 단위

| 구성 요소 | 설명 |
| --- | --- |
| **Owner** | 소유자 정보 |
| **Key** | 객체 이름 (경로 역할) |
| **Value** | 파일 데이터 자체 |
| **Version ID** | 파일 버전 식별자 (버전 관리 활성화 시에만 생성) |
| **Metadata** | 파일 관련 정보 (기본 + 사용자 정의 가능) |

### 객체 URL 예시

```
https://sgu-202500-s3.s3.ap-northeast-2.amazonaws.com/test/haeri.txt
```

- S3 URI: `s3://sgu-202500-s3/test/haeri.txt`
- ARN: `arn:aws:s3:::sgu-202500-s3/test/haeri.txt`

---

## 4. S3 내구성 & 가용성

> **내구성**: 파일을 잃어버리지 않을 능력
> 
> 
> **가용성**: 원할 때 그 파일을 사용할 수 있는 능력
> 
- 내구성: **99.999999999%** (파일 손실 확률 극히 낮음)
- 가용성 SLA: **99.99%** (S3 Standard 기준)

### SLA(Service Level Agreement) 비교

| SLA | 연간 최대 다운타임 |
| --- | --- |
| 99.99% | 약 52분 이하 |
| 99.9% | 약 8시간 45분 이하 |
| 99.5% | 약 1일 18시간 이하 |

---

## 5. S3 스토리지 클래스

용도와 비용에 따라 다른 클래스 선택 가능. **수명 주기 정책**으로 자동 전환도 가능.

### 클래스 한눈에 보기 (비용 높음 → 낮음)

```
S3 Standard → S3-IA → S3 Intelligent → S3 One Zone-IA
→ S3 Glacier Instant → S3 Glacier Flexible → S3 Glacier Deep Archive
```

### 주요 3가지 클래스 비교

| 구분 | S3 Standard | S3-IA | S3 Glacier |
| --- | --- | --- | --- |
| 특징 | 자주 액세스하는 데이터 | 액세스 빈도 낮은 데이터 | 거의 액세스하지 않는 데이터 |
| 사용 사례 | 주요 데이터, 빅데이터 분석, 임시 백업 | 파일 동기화, 데이터 백업, 재해 복구 | 장기 보존, 시스템 백업 |

### 전체 클래스 상세 비교

| 클래스 | 내구성 | 가용성 | AZ 수 | 최소 보관 기간 | 사용 사례 | 주의 사항 |
| --- | --- | --- | --- | --- | --- | --- |
| S3 Standard | 11 9's | 99.99% | 3개↑ | 없음 | 자주 액세스하는 데이터 | - |
| S3 Intelligent-Tiering | 11 9's | 99.99% | 3개↑ | 없음 | 액세스 패턴 예측 불가 데이터 | 객체별 모니터링, 자동 요금 최적화 |
| S3 Standard-IA | 11 9's | 99.99% | 3개↑ | **30일** | 수명 길고 자주 액세스하지 않는 데이터 | 객체당 검색 비용 발생 |
| S3 One Zone-IA | 99.999999% | 99.95% | **1개** | 30일 | 중요도 낮고 자주 액세스하지 않는 데이터 | 단일 AZ, 낮은 내구성 |
| S3 Glacier | 11 9's | 99.99%(복원 후) | 3개↑ | **90일** | 장기 보관 (백업, 아카이브) | 즉시 액세스 불가, 복원 지연 |
| S3 Glacier Flexible | 11 9's | 99.99%(복원 후) | 3개↑ | 90일 | 장애 복구용, 백업 | 분~시간 단위 복원 소요 |
| S3 Glacier Deep Archive | 11 9's | 99.99%(복원 후) | 3개↑ | 90일 | 법적 보관 의무 데이터 | 복원까지 **최대 12시간 이상** |
| S3 on Outposts | - | - | 온프레미스 | - | 온프레미스 환경 데이터 저장 | S3 SDK 및 IAM 연동 필요 |

## 6. S3 보안

- 기본적으로 모든 객체는 **Private(비공개)** 상태로 생성됨
- 퍼블릭 접근은 **명시적으로 허용**해야 함 (예: 정적 웹 호스팅)

### IAM 정책 종류

| 정책 유형 | 설명 |
| --- | --- |
| **Identity-based Policy** (자격 증명 기반) | IAM 사용자, 그룹, 역할에 부여 → "누가 무엇을 할 수 있는가" |
| **Resource-based Policy** (리소스 기반) | S3 버킷 등 리소스에 직접 부여 → "이 리소스에 누가 접근 가능한가" |

### 버킷 정책(Bucket Policy)

- 버킷 단위로 적용되는 **리소스 기반 정책**
- 언제, 어디서, 누가, 무엇을, 어떻게 할 수 있는지 정의
- 익명 사용자 또는 다른 계정에 대한 권한 부여도 가능

**퍼블릭 읽기 허용 예시:**

```json
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::demo.rwlecture.com/*"
}
```

> `arn:aws:s3:::my-bucket/images/*` → 해당 경로의 **모든 객체**에 권한 적용
